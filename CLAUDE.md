## What This Is

**Slidev Loop**: an open-slide-style feedback loop for Slidev. Users comment on rendered slide elements in the browser (dev mode); an AI agent reads the comments, edits `slides.md`, and HMR shows the result. Comments are a **file protocol** (`.slidev/comments.json` + `slides.md`, plain text on disk), deliberately not bound to any single agent — Claude Code, Codex, Cursor, Gemini CLI, and Copilot are all targets.

## Architecture

Three packages with strict separation of concerns (full detail in [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)):

- **`packages/addon`** — the npm package `slidev-addon-loop`, a standard Slidev addon (keywords `slidev-addon` + `slidev`; ship raw `.vue`/`.ts`, Slidev compiles them — no bundling).
  - `global-top.vue`: browser-side comment overlay. Persistent across slides, gated by `import.meta.env.DEV` (absent from builds) and an explicit comment-mode toggle (hotkey, zero interference when off). Gets page number from `$nav.currentPage`.
  - `vite.config.ts` + `plugin/`: dev-server-only Vite plugin (`apply: 'serve'`) serving `/__agent/comments` (GET/POST/DELETE). Writes to `<project root>/.slidev/comments.json` with write-temp-then-rename atomic writes and schema validation.
- **`packages/agent-instructions`** — the single source of truth for workflow instructions (`create-deck.md`, `apply-comments.md`), pure agent-agnostic markdown.
- **`packages/agent-adapters`** — thin per-agent dispatch formats (Claude Code plugin, AGENTS.md, .cursor/, GEMINI.md, copilot-instructions), **generated by a sync script** from agent-instructions. Never hand-write logic in adapters; never let adapter content drift from the canonical source.
- **`playground/`** — local test deck; mounts the addon via relative path (`addons: ['../packages/addon']`).

### Comment data model

`comments.json` entries carry `slideNo` (hard locator — slides are split by `---` surrounded by blank lines, headmatter doesn't count as a page), `elementText` (soft locator — the LLM matches it within that single slide's source), `rect` as 0–1 relative ratios, and `status: open | applied | skipped`. Agents mark comments rather than delete them; UI shows only `open`.

### Key design decisions (do not revisit casually)

- **No per-element source mapping** — page number + element text is enough for an LLM to locate a target within a ~30-line slide. Deliberate non-goal.
- **No WYSIWYG canvas editor**; style changes are expressed via UnoCSS classes or per-slide frontmatter, never inline styles.
- **Don't rebuild Slidev syntax tutorials** — reference existing mature skills (antfu's official slidev skill, marcoshaber99/slidev-skills).
- **MCP server is Phase 5 roadmap, not MVP** — file protocol covers all CLI agents.
- Ambiguous or unlocatable comments are marked `skipped` with an explanation — agents must not guess.

## Development Conventions

- Phase 1 (store + middleware) is explicitly TDD: write Vitest tests first. Required test cases are listed per-file in [docs/TASKS.md](docs/TASKS.md).

Exact floor to be confirmed by Phase 0; record the version used in spikes. Optionally add `@slidev/cli` to peerDependencies as an npm-level signal, but `engines.slidev` stays.
